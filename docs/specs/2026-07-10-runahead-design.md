# runahead — 투기적 병렬 실행 기반 코딩 에이전트 오케스트레이터

> *run-ahead execution: 분기가 해소되기를 기다리지 않고 앞서 실행하는 것.*

작성일: 2026-07-10

---

## 1. 문제

코딩 에이전트는 작업을 하나 끝내면 멈춰서 사람에게 묻는다. "다음에 뭘 할까요?"

사람은 그 순간 자리에 없다. 커피를 마시러 갔거나, 회의에 들어갔거나, 다른 일을 하고 있다. 에이전트는 그동안 아무것도 하지 않는다. 사람이 돌아와 "테스트 짜줘"라고 말하는 순간부터 다시 시계가 돈다.

낭비되는 것은 에이전트의 시간이 아니라 **사람의 벽시계 시간**이다.

## 2. 핵심 아이디어

사람이 다음에 시킬 법한 행동을 **미리 예측해서 실행해둔다.** 사람이 돌아오면 이미 끝나 있는 결과들 중에서 고른다.

이것은 CPU의 분기 예측이 아니라 **run-ahead execution**에 가깝다. 분기를 해소하는 주체가 기계가 아니라 **느린 오라클인 사람**이기 때문이다. 따라서 이 기법의 본질은 **사람의 사고 시간(latency)을 숨기는 것**이고, 가치는 에이전트의 속도가 아니라 **사람이 자리를 비운 시간에 비례**한다.

### 2.1 바이패스 모드와의 차이

`--dangerously-skip-permissions` 같은 바이패스 모드도 사람 없이 진행한다. 차이는 이것이다.

| | 바이패스 모드 | runahead |
|---|---|---|
| 타임라인 | 하나 | 여러 갈래 |
| 사람의 선택권 | **제거됨** | **연기됨** |
| 실패 시 | 되돌려야 함 | 안 고르면 그만 |

바이패스 모드는 위험을 감수하고 자율성을 산다. runahead는 **자율성을 얻으면서 통제권을 유지한다.** 선택은 사라지지 않고 뒤로 미뤄질 뿐이다.

### 2.2 conductor와의 관계

[conductor](https://github.com/j0j1j2/claude-conductor)는 하나의 과제를 여러 에이전트에게 던지고 결과를 취사선택한다. **한 시점에서 공간 방향으로** 병렬화한다.

runahead는 **같은 줄기를 시간 방향으로** 병렬화한다.

> **시간 축(다음에 무엇을 할 것인가)은 runahead가 정한다.**
> **공간 축(그것을 어떻게 할 것인가)은 conductor에게 위임할 수 있다.**

두 도구는 경쟁하지 않는다. conductor는 runahead의 실행 엔진 중 하나로 꽂힌다 (§7.2). conductor 없이도 runahead는 동작하고, 있으면 경쟁 레인이 강해진다.

## 3. 성패를 가르는 단 하나의 제약

> **선택 비용 < 직접 실행 비용**

투기가 이득이려면 사람이 N개 결과를 훑고 고르는 비용이, 그냥 자기가 1개를 시키는 비용보다 작아야 한다. 그렇지 않으면 runahead는 일을 대신 해주는 도구가 아니라 **고민거리를 N배로 늘려주는 도구**가 된다.

롤백 비용은 걱정할 필요 없다. worktree라서 거의 공짜다. **진짜 비용은 사람의 주의력이다.**

이 제약이 아래 모든 설계 결정의 근거다.

## 4. 목표와 비목표

**목표**
- 비동기 시나리오 최적화: 사람이 자리를 비운 동안 리뷰 큐를 채운다
- 사람이 돌아왔을 때 **리뷰 큐가 점점 짧아진다** (개인화의 체감 형태)
- 에이전트 중립: Claude Code, Codex, conductor를 어댑터로 교체 가능
- 학습 데이터는 전적으로 로컬. 서버로 나가지 않는다

**비목표**
- 동기 시나리오(사람이 옆에서 지켜보는 동안의 체감 지연 제거)는 v1 범위 밖이다
- 비가역 행동(push, 배포, 마이그레이션)의 자동 수행은 **영구히** 비목표다
- 파인튜닝, 임베딩, 벡터 DB 없음 (§6.3)

## 5. 핵심 개념

### 5.1 행동 (Action)

runahead의 유일한 단위. 하나의 행동은 하나의 프롬프트를 격리된 worktree에서 실행하고, **하나의 패치**를 낳는다.

브랜치가 아니라 행동이 저장·수락·학습의 단위다.

```
Action {
  kind:      "write-tests" | "add-edge-cases" | "draft-commit" | ...
  prompt:    string
  worktree:  path
  result:    Patch | Failure
}
```

### 5.2 두 개의 레인

| | 고정 레인 | 예측 레인 |
|---|---|---|
| 성격 | 직교적 (orthogonal) | 경쟁적 (competing) |
| 예시 | 테스트 · 린트 · 커밋 초안 · 문서 | 엣지케이스 · 에러 핸들링 · UI 보강 |
| UI | 체크박스 | 라디오 버튼 |
| 서로 | 독립적으로 수락 가능 | 하나만 살아남음 |
| 선택 비용 | ≈ 0 (전부 수락이 기본값) | 비쌈 (읽고 비교해야 함) |
| 사람의 부담 | **덧셈** | **곱셈** |

**두 축을 혼동하면 안 된다.**

- **폭 (breadth)** = 서로 다른 행동의 개수. 직교. 부담은 덧셈.
- **변형 (variants, M)** = 같은 행동의 다른 구현. 경쟁. 부담은 곱셈.

사람에게 A×B 조합표를 제시하는 것은 §3의 제약을 정면으로 위반한다. 사람은 **조합을 설계하지 않는다. 줄 단위로 예/아니오만 한다.** 조합은 시스템이 rebase로 만든다.

### 5.3 확신도 `p` — 단일 다이얼

`p`는 `(직전 작업 유형, 행동)` 쌍에 대해 "사람이 이 행동을 원할 확률"이다. `p` 하나가 세 개의 다이얼을 동시에 돌린다.

| `p` | 변형 수 M | 자식 확장(깊이) | 자동 수락 |
|---|---|---|---|
| 높음 + 분산 작음 | 1 | 허용 | **예** (고정 레인으로 졸업) |
| 중간 | 1 | 허용 | 아니오 |
| 낮음 | 2~3 | 금지 | 아니오 |
| 매우 낮음 | — | — | 애초에 제안 안 함 |

`p`가 오르면 사용자가 체감하는 것은 **깊이가 아니라 폭의 축소**다. 3개 중 고르던 것이 1개만 뜨고, 결국 자동 수락으로 졸업한다. **리뷰 큐가 짧아진다.** 이것이 §3 제약의 해답이다.

### 5.4 졸업과 강등

예측 레인의 행동이 충분히 확신되면 고정 레인으로 **졸업**한다. 확신이 떨어지면 **강등**된다.

이것이 runahead의 진짜 델타다. 손으로 쓴 고정 레시피(`/ship` 류)가 이미 가치의 대부분을 가져간다. runahead는 **그 레시피를 사람이 쓰지 않고 관찰로 발견한다.** 개인화된 `/ship`이 저절로 자란다.

## 6. 예측과 학습

### 6.1 콜드스타트 → 개인화

- **초기**: LLM이 직전 diff와 대화 맥락을 보고 후보 행동을 생성한다.
- **점차**: 사람의 수락 이력이 쌓이면서 후보 선정과 `p`가 개인 통계로 옮겨간다.

### 6.2 학습 신호

세 가지를 기록한다.

1. **miss rate (주지표)** — 사람이 리뷰 큐를 통째로 무시하고 전혀 다른 것을 새로 시킨 비율. 가장 정보량이 큰 신호이며, 예측기가 **놓친** 행동이 무엇인지 알려주는 유일한 출처다.
2. **수락 / 거부** — 각 행동 단위.
3. **rebase 충돌 쌍** — 함께 수락됐는데 머지가 깨진 두 행동. "직교적"은 희망이지 보장이 아니다. 충돌은 버그가 아니라 *"이 둘을 같이 제안하지 마라"* 는 라벨이다.

> **깊이를 여는 트리거는 수락률이 아니라 miss rate여야 한다.**
> 수락률은 *노출된 것들 중의* 수락률이다. 시스템이 자기가 만든 습관 안에서 잘 맞히는 것을 "예측이 좋아졌다"고 착각하면, 확신에 차서 틀린 방향으로 깊이 판다.

### 6.3 학습 = Beta 카운터 두 개

`(작업유형, 행동)` 쌍마다 Beta 분포 하나. 파일 하나로 끝난다.

```json
{
  "feature|write-tests":     { "alpha": 47, "beta": 6  },
  "feature|add-edge-cases":  { "alpha": 12, "beta": 19 },
  "bugfix|write-tests":      { "alpha": 31, "beta": 2  },
  "feature|responsive-ui":   { "alpha": 2,  "beta": 1  }
}
```

세션당 예측이 3~5개뿐이라 그레이디언트를 볼 데이터가 나오지 않는다. 파인튜닝은 오답이다. 여기서 세 가지가 공짜로 딸려온다.

**(1) 탐색이 원리적으로 내장된다.** 후보를 `p̂` 순으로 뽑지 않고 **Beta posterior에서 샘플링**한다 (Thompson sampling). 시도 횟수가 적어 불확실성이 큰 행동은 가끔 높은 값을 뽑아 저절로 큐에 올라온다.

이것이 없으면 **노출 편향**이 시스템을 죽인다: 예측기가 A를 자주 제안 → 사람은 눈앞의 A를 자주 수락 → 통계가 A로 쏠림 → 더욱 A만 제안. 사람이 원래 원하던 B는 **큐에 뜬 적이 없어서 선택될 기회조차 없었다.** Thompson sampling은 별도 장치 없이 이를 해소한다.

**(2) 자동 수락 조건이 정확해진다.** 위 표의 `responsive-ui`는 `p̂ = 2/3 = 0.67`이지만 시도가 3번뿐이다. `47/53 = 0.89`와 같이 취급하면 안 된다. Beta는 평균과 **분산**을 함께 준다.

> 자동 수락 = `p̂`이 높고 **동시에** 분산이 작을 때만.

**(3) 강등이 공짜다.** `β`가 늘면 `p̂`이 내려가고, 알아서 고정 레인에서 예측 레인으로 떨어진다. 별도 로직이 없다.

### 6.4 계층적 사전분포

`P(다음 행동 | 직전 작업)` 을 어디에 조건화할 것인가?

- **전역만** — 데이터는 빨리 모이지만, Rust 레포와 프론트엔드 레포에서 사람의 다음 행동은 전혀 다르다.
- **레포별만** — 정확하지만 새 레포마다 콜드스타트로 돌아간다. "점차 개인화"가 레포 하나 안에서만 성립한다.

**전역 통계를 prior로, 레포별 관측으로 갱신한다.** 새 레포는 사용자의 전역 습관에서 출발해 데이터가 쌓이면 그 레포만의 성격으로 갈라진다. **콜드스타트는 첫 레포에서 딱 한 번만 발생한다.**

## 7. 아키텍처

### 7.1 코어 / 실행 엔진 분리

코어가 소유하는 것 — 이 중 어느 것도 특정 에이전트에 종속적이지 않다.

```
예측기 · 확신도 p · 스케줄러(예산 배분) · 승격/강등
가역성 경계 · 패치 머지 · miss rate 로깅
```

실행 엔진은 인터페이스 하나로 줄어든다.

```
run_action(prompt, worktree) -> Patch | Failure
```

### 7.2 어댑터

| 어댑터 | 역할 | 시점 |
|---|---|---|
| `claude` | headless CLI (`claude -p`) | v1 |
| `conductor` | **경쟁적 변형(M>1) 위임.** 시안 3개를 만들어 라디오 한 줄로 돌려줌 | v2 |
| `codex` | 에이전트 중립성 증명 | v3 |

conductor 통합은 "나중에 갈아탈 대상"이 아니라 **역할 분담**이다. `p`가 낮아 M=3이 필요한 행동을 통째로 conductor에게 던진다. conductor가 없으면 M=1로 낮추면 그만이다.

### 7.3 스케줄링 — 예산을 기대값 순으로

깊이 `D`를 전역 파라미터로 두지 않는다. **확신이 임계를 넘은 노드만 자식을 낳는다.** 확신 높은 경로는 알아서 깊어지고, 낮은 경로는 폭으로만 남는다. best-first 트리 확장이며, 토큰 예산이 자동으로 기대값 높은 가지에 몰린다.

`p = 0.6`이면 깊이 2의 기대값은 `0.36`, 깊이 3은 `0.216`이다. 게다가 2단계는 1단계 위에 쌓이므로 **1단계가 틀리면 그 아래는 전부 쓰레기다.** 비용은 선형으로 늘고 수익은 지수로 준다. 균일 깊이는 오답이다.

### 7.4 정지 조건

**(c) 가역성 경계 — 기본, 자동으로 찍힘**

runahead 전체가 "롤백은 공짜"라는 전제 위에 서 있다. worktree 안의 파일 변경은 실제로 공짜다. `git push`, 배포, 외부 API 호출, DB 마이그레이션은 아니다. **투기가 정당화되는 영역의 경계선이 정확히 가역성이다.**

체크포인트를 사람이 매번 손으로 찍게 하면 §4의 비동기 시나리오 자체가 무너진다. 가역성 경계는 **자동으로 찍히는 체크포인트**다.

따라서 깊이의 천장을 정하는 것은 `p`가 아니라 가역성이다. `구현 → 테스트 → 커밋 → push → PR → 배포` 에서 깊이 4쯤이면 이미 비가역 지대다. **`p`가 0.99가 되어도 `git push`는 하지 않는다.**

**(b) 예산 — 안전장치, v1 필수**

가역적인 일도 무한히 할 수는 없다. 사람이 없는 30분 동안 예측 레인이 스스로를 부풀리며 토큰을 태울 수 있다. headless 에이전트 3~5개를 동시에 돌리면 **레이트 리밋과 토큰이 실제 병목이다.**

```
budget: { tokens: 200_000, wall_clock: 30m, max_actions: 12 }
```

소진되면 정지한다. 나중에 붙일 기능이 아니라 v1의 필수 부품이다.

**(a) 사용자 오버라이드** — 행동 화이트리스트/블랙리스트. 선택 사항.

## 8. 데이터 저장

두 층으로 쪼갠다. **이걸 안 나누면 개인화가 조용히 망가진다.**

```
.git/runahead/              레포 종속 · 임시 · 에이전트가 여기서 실행되지 않음
  patches/                  행동별 패치
  queue.json                현재 리뷰 큐 상태

~/.cache/runahead/          투기가 실제로 실행되는 곳
  worktrees/<repo>/<action>

~/.runahead/                사용자 종속 · 영구 · 절대 커밋 안 됨 · 서버 안 감
  priors.json               전역 Beta 카운터 (§6.3)
  repos/<hash>.json         레포별 Beta 카운터 (전역을 prior로)
  history.jsonl             수락/거부/miss/충돌 원장
```

**worktree는 반드시 `.git` 바깥에 있어야 한다.** 코딩 에이전트에게 `.git` 내부를 작업 디렉토리로 주면 아무것도 편집하지 않은 채 **성공을 반환하고 토큰을 청구한다.** 종료 코드에는 아무 표시가 없고, 빈 패치만이 유일한 증상이다. 실제 에이전트를 붙이기 전까지는 드러나지 않으므로 (페이크 실행기는 파이썬으로 직접 파일을 쓴다) 회귀 테스트로 못 박는다.

작업 트리 안에 두어서도 안 된다. `git status`가 더러워지고, `accept`는 깨끗한 트리를 전제로 한다.

학습 데이터를 레포에 커밋하면 습관 통계가 팀원과 섞인다. 개인화가 아니라 **평균화**다. 동료는 커밋 전에 항상 문서를 쓰고 나는 안 쓴다면, 섞인 통계는 둘 다 틀리게 예측한다.

## 9. 리뷰 큐 — 이 제품의 얼굴

비동기 시나리오에서 성패를 재는 **유일한** 지표다.

```
자동 수락됨 (졸업한 행동)
  ✓ 테스트 작성 + 통과            p̂=0.89 (n=53)   [diff]
  ✓ 린트/타입체크 클린            p̂=0.94 (n=61)   [diff]
  ✓ 커밋 초안                     p̂=0.87 (n=48)   [message]

확인 필요 (예측 레인)
  ☐ 엣지케이스: null 입력 처리     p̂=0.71         [diff]
  ☐ 에러 바운더리 추가            p̂=0.44         ( ) 시안A  ( ) 시안B
  ☐ 모바일 반응형                 p̂=0.31         [diff]   ← 탐색 슬롯

체크포인트에서 정지: git push (비가역)
예산: 토큰 84k/200k · 18분/30분 · 행동 6/12
```

사람은 줄 단위로 예/아니오만 한다. 수락된 행동들은 순서대로 rebase 적용된다.

### 9.1 실패 격리

두 행동을 다 수락했는데 rebase가 깨지면, **그 행동만 실패로 표시하고 나머지는 살린다.** 그리고 그 충돌 쌍을 §6.2의 세 번째 라벨로 기록한다.

## 10. 테스트 전략

`p` 학습과 스케줄링은 LLM 없이 검증 가능해야 한다. 그것이 코어/실행 엔진 분리(§7.1)의 두 번째 이유다.

- **`run_action`을 페이크로 교체**하고 코어 전체를 테스트한다. LLM 호출 없이 결정론적으로.
- **Beta 갱신 · Thompson sampling · 졸업/강등 임계**: 순수 함수. 단위 테스트.
- **노출 편향 회귀 테스트**: 합성 사용자(항상 B를 원하지만 초기 통계는 A에 쏠려 있음)를 시뮬레이션하고, N 세션 안에 B가 큐에 등장하는지 검증한다. Thompson sampling이 실제로 작동하는지 확인하는 유일한 방법이다.
- **예산 소진**: 무한 확장을 시도하는 페이크 예측기를 넣고 상한에서 멈추는지 확인한다.
- **가역성 경계**: `git push`를 포함한 행동이 절대 실행되지 않는지 확인한다. 이것은 테스트가 아니라 **안전 요건**이다.
- **rebase 충돌 격리**: 같은 파일을 건드리는 두 패치를 만들어 하나만 실패하고 나머지가 살아남는지 확인한다.

## 11. 마일스톤

| | 범위 | 완료 기준 |
|---|---|---|
| **v1** | 코어 + `run_action()` 인터페이스 + `claude` 어댑터 + 예산 + 가역성 경계 + 리뷰 큐 CLI | 실제 레포에서 한 사이클: 작업 → 투기 → 큐 → 수락 → rebase |
| **v2** | `conductor` 어댑터 (경쟁 변형 위임) | M>1인 행동이 conductor를 거쳐 라디오 한 줄로 돌아옴 |
| **v3** | `codex` 어댑터 · 계층적 prior가 의미 있는 데이터량 도달 | miss rate가 세션 수에 따라 하락함을 실측 |

순서보다 **경계가 중요하다.** 경계를 처음에 제대로 그으면 v2는 재작성이 아니라 파일 하나 추가다.

## 12. 열린 위험

1. **`p`가 실제로 수렴하는가.** 세션당 3~5 샘플이면 유의미한 개인화까지 수십 세션이 필요하다. v3의 완료 기준(miss rate 하락 실측)이 이 가설의 유일한 검증이다. 수렴하지 않으면 runahead는 §5.4의 델타를 잃고 "손으로 쓴 고정 레시피"로 축소된다.
2. **레이트 리밋.** 병렬 headless 에이전트가 실제로 몇 개까지 가능한지 v1에서 측정해야 한다. `max_actions`의 현실적 상한이 여기서 나온다.
3. **작업 유형 분류.** `p`는 `(작업유형, 행동)`에 조건화되는데, "이번 작업이 feature인가 bugfix인가"를 누가 정하는가. v1은 LLM에게 묻는다. 분류가 흔들리면 통계가 오염된다.
4. **체인 후속 단계의 빌린 증거.** 체인은 통째로 수락되므로 `write-tests → run-tests → draft-commit`을 수락하면 세 행동 모두 `accepted`로 기록된다. 그러나 사람이 실제로 판단한 것은 체인의 루트다. 후속 단계는 독립적으로 심사받은 적 없이 신뢰도를 쌓고, 빌린 증거로 졸업할 수 있다. v1은 이를 허용한다 — 사람이 체인 전체를 받았으니 틀린 기록은 아니다. 그러나 후속 단계가 단독 후보로 제안될 때 그 `p`는 과대평가되어 있다. 관측된다면 체인 내 위치별로 카운터를 분리해야 한다.
